The problem we start from

This problem was proposed to us during cliw lab, you can find more about it here.

Create a MV * mini-framework for the development of vector-driven vector-based web applications at the client level. As an actual case study, it will be used to implement a Tamagotchi game simulator. At the very least, each authenticated user will manage one or more characters he will take care of (nourish, educate, amuse). The Web application will provide an interactive way to create and configure these characters, plus sharing them to users. Bonus: Implementing a level message routing system client to build SPA-Single Page Applications.

Structure

We explained the architecture of our application by taking into account several aspects, in order to fulfill the requirements which can be found here. For this part of the evaluation, we structured our project into the following essential sections:

SPA

A single page application interacts with the user by dynamically rewriting the current page rather than loading new pages. This type of application is popular among the developers because of several advantages, such as avoiding interruption of user experience, although it might be more difficult to create. There are various techniques which can be used when creating such applications: use of some JavaScript frameworks, AJAX, Websockets, Browser plugins.

Modules

The modules will be built according to the functionalities offered by our application. The following list contains the main modules, together with the corresponding submodules.

  • Users management
    • Login
    • Register
    • Password reset
  • Tamagotchi management
    • Create tamagotchi
    • Edit tamagotchi
    • View tamagotchi
    • Remove tamagotchi
    • Share tamagotchi

Login submodule has the role of authenticating the existing users, in order to give them access to all the facilities offered by our application. It has to be provided the user credentials.

Register submodule has the role of creating new accounts, based on the forms submitted by the persons who wish to sign up.

Password reset submodule handles the requests from users who need a new password in order to login.

Create tamagotchi submodule has the role of inserting new tamagotchies into the database, by using the data provided by the authenticated users. The data consists of features selected by the users for the new tamagotchies, such as color or costume.

Edit tamagotchi submodule is similar to the previous one, but the users need to select an existing tamagotchi and modify some of its features. The characteristics of the tamagotchi will be updated in the database.

View tamagotchi submodule lets the users visualise a selected tamagotchi and perform several actions such as feeding or entertaining.

Remove tamagotchi submodule lets the users delete a tamagotchi from their account in the database.

Share tamagotchi submodule makes possible the sharing among users of their tamagotchies, by using a link.

The modules/submodules interact with each other in different ways, depending on the use-cases. However, there is a permanent dependence between the two main modules - the users management module decides, by whether authenticating a user or not, which actions can be performed by the tamagotchi management module for a certain user.

The functions of each module, the way they interact with each other/the user and the way the user can access each function in the interface are detailed in the following sections, using diagrams and written explanations when necessary.

Database Representation and Data Flow

Description of what is in the diagrams

Regarding the data exchange in our application, the submodules that are in charge of the users management interact mainly with the database because it keeps track of the existing users and their accounts, but also with mail servers, due to the fact that every account requires an email address at creation time.

Who wishes to register first needs to complete a form with some personal information, including an email address. The data is verified (especially the validity of the email address). If something is wrong the user is notified, otherwise he/she gets a confirmation email. After the user confirms, an account is created and inserted into the database, and now the user can enjoy the facilities of our application.

An existing user may login into our application by entering his/her credentials. The login process queries the database and if the requested account is found and the password is correct, the user is granted access into the application, otherwise an error is returned.

The password can be reset at the user's request, if he/she has the correct email address corresponding to the account. The email address and the associated account are verified in the database. If the user didn't enter the correct email address, the process fails, otherwise an email is sent to the address. If the password reset request is confirmed, a new password is generated and given to the user.

All the data flows described in the above paragraphs can be visualised in the Authentication diagram.

An authenticated user can enjoy the facilities of our application. The tamagotchi management modules intermediate the transfer of data between the user and the database, considering that the database is storing the properties of the tamagotchies created by the users and also some animations which make the tamagotchi more fun.

When a user wants to create a new tamagotchi, he/she needs to set its properties. Some of them are mentioned below:

  • name
  • face shape - possible values are based on geometric figures
  • eyes shape - the setting of this parameter is done just like in the case of the face shape
  • mouth - for a plus of expressivity, the mouth can be adjusted in order to reflect the character's feelings
  • costume - the user can choose the character's clothes, from a set of predefined items
  • accesory - cool accesories like hats or helmets are provided in addition to the costume
  • face color

When the user presses the save button, the data is collected in order to be used at tamagotchi creation. After the new tamagotchi is created and inserted into the database it becomes available to the user.

Editing is similar, except the user modifies an existing tamagotchi. Therefore, the corresponding tamagotchi in the database will be updated with the new data collected from the user.

When a user needs to view a tamagotchi, the application queries the database and after the tamagotchi is found, it will be loaded in the view page.

Tamagotchies can also be removed from the database, at the user's request.

Another facility is the sharing of tamagotchies among users via a link.

The data flow described in the above paragraphs can be seen in the Tama Management diagram.

External resources/tools

For the creation of the svg drawings and data storage, we used the following resources/tools.

Adobe Illustrator

Adobe Illustrator is a vector graphics editor created and distributed by Adobe Systems. It was originally developed for Apple Macintosh computers (now it works on more operating systems) and among the years it had several versions. Some of its basic tools are selection, paintbrush, pen, pencil tools. More information about the soft and the pricing details for clients in Romania can be found here.

We will use the tool for creating the svg figures that are necessary when the applications displays a tamagotchi (for the animations we will make more frames). The SVG figures will be injected into the code.

Firebase Realtime Database

Firebase Realtime Database is a NoSql cloud database developed by Google. It has several features, like data syncronization in realtime, the ability to work even offline (when the connection is reestablished the client gets the changes from the server) or scaling. More details can be found here.

We will use the database to keep the track of the existing users, their tamagotchies, and also some animations associated with the tamagotchies.
